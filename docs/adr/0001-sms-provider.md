# ADR-0001 — مزوّد الرسائل النصية (SMS Provider) لرموز الدخول

- **الحالة:** مقترح (Proposed) — قرار تقني بانتظار اختيار الفريق الهندسي.
- **النوع:** قرار معماري/تقني (نُقل من سجل القرارات الاستراتيجية بقرار تنفيذي، لأنه ليس قرار حوكمة).
- **التاريخ:** ٢٤ يوليو ٢٠٢٦.
- **المبادئ الحاكمة:** م-١١ (ندمج لا نبني)، هـ-٢٢ (حماية الدخول + حدّ معدّل).

## السياق
الدخول يعتمد Supabase Auth (Phone OTP). في البيئة الحيّة اليوم يُستخدم **رقم اختبار برمز ثابت وبيانات Twilio وهمية** — فلا يستلم أي مستخدم حقيقي رمزاً. هذا **حاجب لأول عميل حقيقي (Beta)**. المطلوب مزوّد رسائل سعودي عبر **Supabase Send-SMS Hook (Edge Function)**.

## الخيارات
1. **Unifonic** — واسع الانتشار في المملكة، API ووثائق أقوى، دعم Sender-ID، شائع لـ OTP.
2. **Taqnyat** — محلّي، تسعير تنافسي، تكامل مباشر.
3. مزوّد Supabase المدمج (Twilio/MessageBird/Vonage) — أبسط تقنياً لكنه أقل ملاءمة تنظيمية/سعرية للسوق السعودي.

## التوصية
البدء بـ **Unifonic** كافتراضي (ergonomics أقوى)، مع **Taqnyat** بديلاً. التنفيذ عبر Send-SMS Hook دون تغيير نموذج المصادقة.

## التبعات
- تسجيل **Sender ID** له مهلة تنظيمية (CITC) — يبدأ مبكراً.
- يتطلّب فتح حساب لدى المزوّد + مفتاح API يُحفظ كسرّ (لا يُلتزَم — هـ-١٩).
- يفتح مسار Beta وأول عميل.

## القرار المطلوب من الفريق الهندسي
اختيار المزوّد (Unifonic/Taqnyat)، وفتح الحساب، وبدء تسجيل Sender ID. يُحدَّث حقل «الحالة» إلى «مُعتمد» عند الاختيار.
